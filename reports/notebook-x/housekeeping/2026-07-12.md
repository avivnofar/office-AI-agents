### 1. Implement "Public Index" Validation Middleware
Since `notebooks/_index-public.json` is now a public API contract for the *Data Center* project, any malformed write to this file will break the external integration.
*   **Action:** In `api_server.py` or `github_storage.py`, create a validation function that runs before any `github_put` operation targeting `_index-public.json`.

If the schema validation fails, the API should return a `422 Unprocessable Entity` and abort the write to prevent corrupting the Data Center’s dependency.

### 2. Add "Data Quality" State Machine
The context mentions that notebooks only appear in the Data Center once they reach `partial` or `complete` status. Currently, this seems to be a manual or implicit process.
*   **Action:** Formalize the `dataQuality` field in the notebook metadata schema.
*   **Detail:** Update the `notebook_backend.py` logic to include a `transition_quality(notebook_id, new_status)` function. This function should enforce that a notebook cannot move to `complete` unless it contains all four autoseed files (Core Concepts, Cheat Sheet, Troubleshooting Labs, Interview Prep). This ensures that the public-facing index only promotes high-quality, fully-formed notebooks.

### 3. Implement an "Export-to-Agent" Bridge Endpoint
The context identifies a missing link between the `/api/notebooks/{name}/export` endpoint and the *Office AI Agents* system.
*   **Action:** Create a new endpoint `/api/notebooks/{name}/agent-ready` that returns a sanitized, token-optimized version of the notebook.
*   **Detail:** This endpoint should strip unnecessary metadata and return a structured JSON object containing the `combined_markdown` and a `metadata_summary` field. This provides a clean, predictable interface for the 11-agent system to consume without needing to parse the full raw notebook structure.

### 4. Add "Session-Aware" GitHub Sync Logging
With the introduction of `github_storage.py` and `data_safety.py`, debugging failed writes to GitHub is likely difficult.
*   **Action:** Add a simple `audit_log.json` file to the GitHub repository that tracks the last 10 write operations.
*   **Detail:** Every time `validate_write()` in `data_safety.py` passes, append a small entry to `audit_log.json` containing the timestamp, the file modified, and the `dataQuality` state of the notebook. This creates a "black box" recorder that allows you to trace why a specific notebook might have disappeared from the Data Center index due to a failed sync or overwriting issue.

### 5. Automate "Index-Public" Refresh on Autoseed
The current workflow relies on manual updates to the public index.
*   **Action:** Update the autoseed completion flow in `notebook_backend.py` to trigger an automatic update of `_index-public.json`.
*   **Detail:** When the four autoseed files are successfully generated, the backend should automatically append the new notebook entry to the index with a default `dataQuality` of `partial`. This reduces the friction of "publishing" a new notebook and ensures that the Data Center is updated as soon as the generation process finishes.

### 1. `notebook_backend.py`: Incomplete `create_new_notebook`
The excerpt shows the `create_new_notebook` function is cut off mid-exception handling.

### 2. `api_server.py`: Missing `await` in `add_url_to_notebook`
*   **The Bug:** The endpoint `add_url_to_notebook` calls `nb.add_url_content`, which likely involves network I/O (fetching the URL). However, the route is defined as `def` (synchronous) rather than `async def`, and it does not `await` the call.
*   **The Fix:** Change the route definition to `async def add_url_to_notebook(...)` and update the call to `res = await nb.add_url_content(...)`.

### 3. `api_server.py`: Inconsistent Error Handling for `export_notebook`
*   **The Bug:** In `export_notebook`, you check `res.get("status") == "error"`. However, in other endpoints (like `list_notebook_files`), you check `res["status"] == "error"`. If `res` does not contain the key "status" (e.g., if the backend returns a raw object or a different error structure), this will raise a `KeyError` instead of returning a clean 400 error.
*   **The Fix:** Standardize the backend return format. Ensure every backend function returns a dictionary containing at minimum a `"status"` key. In the API layer, use `res.get("status")` consistently across all endpoints to avoid `KeyError` crashes.

### 4. `notebook_backend.py`: Potential Race Condition in `get_all_notebooks`
*   **The Bug:** `get_all_notebooks` checks `os.path.exists` and then calls `os.listdir`. In a high-concurrency environment, the directory could be deleted between these two calls, causing an `OSError`.
*   **The Fix:** Wrap the `os.listdir` call in a `try-except OSError:` block to gracefully handle cases where the directory disappears during execution.

### 5. `api_server.py`: Missing `await` in `ask_knowledge_notebook`
