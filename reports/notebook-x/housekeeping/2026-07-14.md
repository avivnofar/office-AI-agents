### 1. Implement "Data Quality" Validation Middleware
Currently, the `dataQuality` field is critical for the `data-center` integration, but it appears to be a manual or loosely managed attribute.
*   **Action:** Add a validation function in `data_safety.py` that checks the content of the 4 autoseeded files (Core Concepts, Cheat Sheet, Troubleshooting Labs, Interview Prep) before allowing a notebook to be marked as `complete`.
*   **Implementation:** Create a `validate_notebook_completeness(notebook_data)` function that scans for empty files or placeholder text. If the notebook is missing one of the four required files, force the `dataQuality` to `draft` or `partial` automatically during the `github_put` operation. This prevents "broken" notebooks from being indexed by the Data Center worker.

### 2. Automate `_index-public.json` Sync on Write
The `data-center` integration relies on `notebooks/_index-public.json`, but there is no mention of an automated sync process when a notebook is created or updated.
*   **Action:** Update `github_storage.py` to trigger an index refresh whenever a notebook is saved.
*   **Implementation:** Create a `sync_public_index()` function in `notebook_backend.py`. This function should fetch the current `_index-public.json`, iterate through the existing notebooks in the GitHub repo, rebuild the array with the updated metadata (including the `dataQuality` field), and push the updated JSON back to GitHub. Call this function at the end of every successful `github_put`.

### 3. Add "Export for Agent" Endpoint
The context notes that the `/api/notebooks/{name}/export` endpoint is the logical hook for the `office-AI-agents` system, but it is currently unused.
*   **Action:** Standardize the export payload to include a "System Prompt" snippet.
*   **Implementation:** Modify the export endpoint in `api_server.py` to return a JSON object containing the `combined_markdown` *and* a pre-formatted `system_prompt_snippet` (e.g., "You are an expert on [Notebook Name]. Use the following context: [Truncated Markdown]"). This makes it trivial for the 11-agent system to consume the notebook without needing to parse the raw markdown itself.

### 4. Implement a "Health Check" for GitHub Persistence
Since the app relies on GitHub as the primary storage layer, a failure in `github_storage.py` could lead to data loss if the frontend assumes a save was successful.
*   **Action:** Implement a "Write-Verify" pattern in `data_safety.py`.
*   **Implementation:** After a `github_put` operation, the backend should perform a lightweight `github_get` (or check the ETag/commit hash) to confirm the file exists and matches the expected content size. If the verification fails, the API should return a `503 Service Unavailable` or `500 Internal Server Error` to the frontend, triggering a "Save Failed - Retry" UI state in `index.html` rather than silently failing.

Based on the provided excerpts, here are the identified issues and recommended fixes:

### 1. `notebook_backend.py`: Incomplete `create_new_notebook`
The excerpt shows the `create_new_notebook` function is cut off mid-exception handling.
*   **The Bug:** The function lacks a `try-except` block completion and, more importantly, lacks the logic to actually create the directory and metadata file.
*   **The Fix:** Ensure the function completes the `_resolve_notebook_path` call, then uses `os.makedirs(notebook_path)` to create the folder. It should also write the `metadata` dictionary to a `metadata.json` file inside that new directory to ensure the notebook state is persisted.

### 2. `notebook_backend.py`: `call_gemini` Timeout/Fallback Logic
*   **The Bug:** The `call_gemini` function uses a 180-second timeout. While this is generous, the fallback to `_call_groq` only triggers on `httpx.RequestError` or specific status codes (429/503).
*   **The Fix:** Add a check for `500` (Internal Server Error) status codes from Gemini. Often, when the Gemini API is unstable, it returns a 500 rather than a 503. Adding `500` to the fallback trigger list will significantly improve the reliability of the "Notebook-X" assistant.

### 3. `api_server.py`: Missing Error Handling in `ask_knowledge_notebook`
*   **The Bug:** The `ask_knowledge_notebook` endpoint is truncated and lacks the `if res.get("status") == "error":` check present in other endpoints.
*   **The Fix:** Implement the same error handling pattern used in `ask_gemini` or `create_notebook`. If `nb.ask_knowledge_notebook` returns a status of "error", the API should raise an `HTTPException` with the appropriate status code (e.g., 400 or 500) rather than potentially returning a malformed response or crashing.

### 4. `api_server.py`: Potential Race Condition in `lifespan`
*   **The Bug:** The `lifespan` context manager calls `nb.sync_from_github()` synchronously. If the GitHub repository is large, this will block the FastAPI server from starting (or responding to health checks) for an extended period.
*   **The Fix:** If `sync_from_github` involves network I/O, consider wrapping it in a background task or ensuring it has a strict timeout. At minimum, add a `try-except` block around the startup sync so that a network failure during startup doesn't prevent the entire API server from booting.

### 5. `notebook_backend.py`: Security/Path Traversal
*   **The Bug:** While `_resolve_notebook_path` checks for path separators, it relies on `os.path.abspath`. On some systems, if `BASE_DATA_DIR` is not correctly normalized, there might be edge cases with symlinks.
*   **The Fix:** Use `pathlib.Path` for all path manipulations. `Path(notebook_path).resolve()` is more robust than `os.path.abspath` for preventing directory traversal attacks.

### 6. `api_server.py`: Missing `Content-Type` validation on `import_notebook`
*   **The Bug:** The `import_notebook` endpoint reads the file and attempts `json.loads`. If a user uploads a non-JSON file (like a large binary), `json.loads` will raise a `JSONDecodeError`, which is currently caught by a generic `Exception` block.
*   **The Fix:** Explicitly catch `json.JSONDecodeError` to provide a specific "Invalid JSON format" error message to the user, rather than a generic 400 error.
