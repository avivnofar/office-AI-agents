## General recommend-changes pass

Based on the `CLAUDE_CONTEXT.md` and the current architectural state of Notebook-X, here are 4 concrete, actionable improvements to strengthen the system:

### 1. Implement "Data Quality" Validation Middleware
Currently, the system relies on manual updates to `dataQuality` for the `_index-public.json` contract. To prevent breaking the Data Center integration, add a validation check in `data_safety.py` (or a new `schema_validator.py`) that runs before any `github_put` operation on the index file.
*   **Action:** Create a function `validate_notebook_schema(data)` that asserts the presence and type of `id`, `name`, `domain`, `dataQuality`, `summary`, and `githubRawUrl`. If `dataQuality` is set to `complete`, enforce that `summary` must be at least 100 characters. Reject the write if the schema is invalid to ensure the Data Center worker never receives malformed JSON.

### 2. Automate `_index-public.json` Sync via Lifespan Hook
The context mentions that `api_server.py` has a `lifespan` startup sync. Currently, it is unclear if this sync is robust enough to handle the new public index contract.
*   **Action:** Update the `lifespan` logic in `api_server.py` to perform a "Consistency Check" on startup. It should scan the `/notebooks/` directory, compare the local state against the `_index-public.json` on GitHub, and automatically generate/patch missing entries for any notebooks that exist in storage but are absent from the index. This ensures that new notebooks are "born" into the public API contract immediately.

### 3. Implement "Export-to-Agent" Payload Sanitization
Since the `combined_markdown` export is the intended hook for the `office-AI-agents` system, it currently lacks a standard format for agent consumption.
*   **Action:** Modify the `/api/notebooks/{name}/export` endpoint in `notebook_backend.py` to inject a standard "System Header" at the top of the `combined_markdown` string. This header should include the notebook's metadata (Name, Domain, Data Quality) in a structured YAML front-matter block. This allows the 11-agent system to parse the context automatically without needing custom logic for every notebook type.

### 4. Add a "Dry Run" Mode for Autoseed
The autoseed feature generates four authority-backed files, but if the Gemini prompt fails or produces low-quality output, it currently commits directly to the file system/GitHub.
*   **Action:** Introduce a `preview` flag to the autoseed API route. When `preview=true`, the backend should return the generated markdown content to the frontend UI for user review *before* calling `github_put`. Add a "Commit" button in the UI that only triggers the actual file write after the user has verified the generated content. This protects the `dataQuality` integrity of the notebooks.

Based on the provided excerpts, here are the identified issues and recommended fixes:

### 1. `notebook_backend.py`: Incomplete `create_new_notebook`
The excerpt shows the `create_new_notebook` function is cut off mid-exception handling.
*   **The Bug:** The function lacks a `try-except` block completion and, more importantly, lacks the logic to actually create the directory and save the `metadata.json` file.
*   **The Fix:** Ensure the function completes the directory creation using `os.makedirs(notebook_path, exist_ok=True)` and writes the `metadata` dictionary to a `metadata.json` file inside that directory using `json.dump()`. Ensure it returns a dictionary with `{"status": "success"}` or `{"status": "error", "message": "..."}` to match the API expectations.

### 2. `api_server.py`: Missing `await` in `add_url_to_notebook`
*   **The Bug:** The endpoint `add_url_to_notebook` is defined as a standard `def` function, but it calls `nb.add_url_content`. If `add_url_content` performs network requests (which it likely does, given the name), it should be an `async` function.
*   **The Fix:** Change the endpoint definition to `async def add_url_to_notebook(...)` and `await` the call to `nb.add_url_content(...)`.

### 3. `api_server.py`: Inconsistent Error Handling
*   **The Bug:** In `ask_gemini`, you catch all exceptions and return a `500` error. However, in other endpoints (like `create_notebook`), you rely on the backend returning a `{"status": "error"}` dictionary.
*   **The Fix:** Standardize the pattern. Either have the backend raise custom exceptions (e.g., `NotebookNotFoundError`) and use a FastAPI Exception Handler, or ensure every backend function returns the `{"status": "error", "message": "..."}` structure. Currently, `ask_gemini` might return a 500 for a user-input error that should be a 400.

### 4. `notebook_backend.py`: Potential Race Condition in `get_all_notebooks`
*   **The Bug:** `get_all_notebooks` checks `os.path.exists` and then `os.listdir`. In a high-concurrency environment, the directory could be deleted between these two calls.
*   **The Fix:** Wrap the `os.listdir` call in a `try-except FileNotFoundError` block to handle cases where the directory is removed by another process or a cleanup task between the check and the read.

### 5. `api_server.py`: Missing `await` in `ask_knowledge_notebook`
*   **The Bug:** The excerpt shows the `ask_knowledge_notebook` function is cut off inside the `except` block.
*   **The Fix:** Ensure the `except` block returns a proper `HTTPException` rather than just letting the error bubble up or returning `None`. 
    *   *Example:* `raise HTTPException(status_code=500, detail=f"Knowledge retrieval failed: {str(e)}")`

### 6. Security/Safety Note
*   **The Observation:** `_resolve_notebook_path` correctly prevents directory traversal using `startswith`, but `_validate_notebook_name` allows names that might conflict with system files on certain OSs (e.g., `CON`, `PRN` on Windows).
