# Project Audit Document

## 1. Architecture & Model Configuration

* **Current Config Discrepancy**: The `config/agents-config.json` currently assigns Gemini as the default model provider for all agents (1 through 11).
* **Intended Model Distribution**:
  * **Lower-Level Agents (Standard Tier - Agents 1-4)**: Should be routed to **Groq** to leverage low-latency, cost-efficient inference for high-frequency operations.
  * **Admin/Specialist Agents (Sudo/Root Tiers - Agents 5-11)**: Should continue using **Gemini** to utilize its larger context windows and superior reasoning capabilities for strategic decision-making, planning, and asset generation.
* **Cloudflare Worker Dual-Instance Review**: The Cloudflare Worker currently initializes two distinct Gemini instances—one dedicated to handling overall project/workspace context and another handling the dynamic agent simulation. These instances must be audited to prevent rate-limiting conflicts, ensure proper API key isolation, and optimize state management between the two runtimes.

## 2. Critical Bugs

* **The Issue**: An intermittent but severe `"Unterminated string in JSON"` syntax error occurs in `notebook-x-daily.mjs` during the execution of the `frontend_code_change` step.
* **Root Cause**: The step relies on strict `JSON.parse()` to extract raw frontend code from an LLM response. When the model outputs unescaped quotes, block indents, or raw markdown code fences, the JSON structure breaks, throwing a fatal parsing exception.
* **The Solution**:
  1. Deprecate strict `JSON.parse()` for extracting multi-line block payloads (such as frontend code).
  2. Implement a robust **Tag/Regex-based extraction** pattern. Have the model wrap its output in explicit XML-like boundary tags (e.g., `<code_payload>...</code_payload>`).
  3. Extract the inner string using a stable regex wrapper that processes the content as raw text, bypassing JSON escaping vulnerabilities entirely.

## 3. Passover Cleaning

To restore order and modularity to the current workspace, the following directory reorganization steps should be executed:

* **Consolidate Configuration Files**: Move all environment, schema, and operational configs scattered across the root into the unified `/config/` directory (e.g., `agents-config.json`, `asset-platforms.json`, and any environment overrides).
* **Modularize the Core Source**: Establish a clean `/src` structure:
  * `/src/agents/`: Group `agent-base.js`, `agent-runner.js`, and individual agent sub-classes.
  * `/src/services/`: Group shared API engines, databases, and LLM provider connectors (Gemini/Groq adapters).
  * `/src/utils/`: Place helpers, system prompts, and formatters here.
* **Isolate Simulation and Archive Artifacts**: Standardize dynamic outputs into dedicated, git-ignored data directories:
  * `/data/sessions/` for active simulation run-logs.
  * `/data-center-archive/` strictly for persistent, committed files such as generated markdown guides and incident reports.
* **Prune Root-level Utility Scripts**: Move root-level automation scripts (e.g., `notebook-x-daily.mjs`, test runners, and seed scripts) into a `/scripts/` or `/tasks/` directory to keep the root workspace clean and maintainable.
## 4. D1 Database & State Management

* **Database Schema Analysis**: The Cloudflare D1 instance serves as the persistent state engine for the entire simulation framework. Based on the operational requirements of the workers and orchestration layers, the relational schema comprises the following primary tables:
  * `agents`: Stores persistent, long-term agent state across runs (e.g., `id`, `name`, `mood`, `irritation_level`, `permanent_flags`, `updated_at`).
  * `agent_sessions`: Tracks single execution environments and boundaries (e.g., `id`, `agent_id`, `cases_handled`, `mood_start`, `mood_end`, `extended_session`, `created_at`).
  * `interactions`: Stores detailed query-response pairs for quality and mood evaluations.
  * `suggestions`: Tracks submitted improvement suggestions routed via the permission tiers (`root`, `sudo`, `specialist`, `standard`).
  * `incident_reports` & `weekly_reports`: Persists administrative and escalation records generated by agents during severe states.
* **State Persistence Lifecycle**: Because Cloudflare Workers run in a stateless serverless environment, memory does not persist across invocations.
  1. **Sync-On-Boot**: When a session starts, the orchestrator queries D1 for the agent's last known state (mood, irritation stack, and permanent flags).
  2. **In-Memory Cycle**: The agent processes its work queue in-memory via its specific class instance (`AgentBase` subclass), updating local flags.
  3. **Sync-On-Exit**: Upon session completion or terminal state triggers (such as `ANGRY` or `TRAINEE_PANIC`), the final state is committed back to D1, closing the `agent_sessions` row.

## 5. Secrets & Environment Variables

* **Secret Bindings & Configuration**: Local development relies on a `.dev.vars` file to store high-privilege keys, while production environments leverage Cloudflare's secure environment bindings. Key variables include:
  * `GEMINI_API_KEY`: High-reasoning access token for admin-tier agents and complex code generation.
  * `GROQ_API_KEY`: Ultra-fast, low-latency API access token for high-frequency operations of agents 1-4.
  * `GITHUB_TOKEN`: Utilized for committing generated guides or documentation changes to `data-center-archive`.
* **Security Best Practices for the New Architecture**:
  1. **Local Isolation**: Never commit `.dev.vars` or any environment overrides to GitHub; ensure they are fully captured in the root `.gitignore`.
  2. **Cloudflare Secret Management**: Bind API keys using `wrangler secret put <KEY>` instead of defining them in plaintext under the `[vars]` block of `wrangler.toml`.
  3. **Runtime Least Privilege**: Ensure that the Cloudflare Worker runtime only passes the specific provider key required for the executing agent's tier to mitigate key leak exposure during execution cycles.

## 6. Frontend & UI Integration

* **API Endpoints & Communication Hub**: The Cloudflare Worker backend exposes concrete interface boundaries for `Notebook-X`:
  * `POST /api/chat`: Main execution endpoint for invoking a specific agent with a prompt or task query.
  * `GET /api/agents`: Retrieves the current state, mood, irritation level, and status of all 11 agents.
  * `GET /api/sessions`: Fetches simulation logs, performance analytics, and session history.
  * `POST /api/suggestions/approve`: Allows operators to review and act on suggestions stored in D1.
* **Strategic Note on Stitch UI Parsing**: To seamlessly integrate with the upcoming user interface designs currently being designed in Stitch, responses must decouple structural rendering from natural-language execution streams.
  * **Regex/Tag Separation**: Rather than outputting mixed JSON/Markdown that forces fragile UI parsing, backend responses should wrap raw modifications, code payloads, or metadata in clear, non-nested tags (e.g., `<code_payload>...</code_payload>` or `<ui_patch>...</ui_patch>`).
  * **Deterministic Extraction**: The Stitch-integrated frontend can then utilize a lightweight, reliable regex engine to intercept these specific block tags, rendering rich visual code components or interactive state widgets on-the-fly, while seamlessly flowing standard conversational text directly into the chat container.
