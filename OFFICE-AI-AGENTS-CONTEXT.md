## 1. What this project is

An 11-agent AI office simulation that trains and stress-tests the Claude
model embedded in a companion app, **data-center** (an IT/cybersecurity
knowledge base). Agents role-play distinct personas, process realistic IT
support cases, hold meetings, and file reports. The office is a **model
training mechanism first** — personality/mood simulation exists to
generate realistic, varied usage patterns that stress-test and improve
the model, not as an end in itself.

## 2. Repo structure

- **office-AI-agents** (this repo) — simulation engine, agent logic,
  configs, D1 database schema, GitHub Actions automation.
- **data-center** (separate repo, `avivnofar.github.io/data-center`) —
  the app agents work on. Claude-Sonnet-5-powered AI Search bar. Scope
  recently expanded from Netvill-specific telecom to **general IT +
  cybersecurity**.

## 3. Execution model

- Runs **one simulated day at a time** — not a continuous year-long
  clock. (Earlier "yearly deliverables" framing in README.md is
  superseded; year-tracker/vacation-day complexity was deliberately
  removed as not serving the training goal.)
- **Nightly path:** GitHub Actions, 02:30 Israel time, Sun–Thu, runs one
  full simulated day end-to-end via a direct Anthropic API session and
  commits a daily report.
- **Daytime path:** Cloudflare Worker cron, every 30 min, 08:00–16:30
  Israel time. Checks `daily-schedule.json` and runs whatever block is
  due (case batches, tool-task windows, standups, spare time). State
  persists between ticks in Cloudflare KV (`SIM_KV`, key
  `daily-cycle-state`).
- **Weekly rhythm:** Sun–Thu full days (5 case batches + tool-task +
  standup + reports) · Friday short day (2 case batches + weekly
  executive summary) · Saturday fully idle (zero API calls).

## 4. Architecture

| Component | Technology |
|---|---|
| Worker | `data-center-agents` (`workers/agent-runner.js`, entry per `wrangler.toml`) |
| Durable Object | `AgentStateDO` (`workers/state-manager.js`) |
| State storage | Cloudflare D1 (`data-center-db`) + KV + Durable Objects |
| Service binding | `APP_API` → `data-center-api` (Workers can't `fetch()` another Worker's `*.workers.dev` URL directly — error 1042) |
| Cron | `*/30 5-13 * * *` UTC = 08:00–16:30 IDT |
| Nightly automation | `.github/workflows/scheduled-claude.yml` → `.github/scripts/run-claude-session.js` + `commit-and-log.sh` — a separate path from the Worker's own cron |

**Runtime consolidation (per STRATEGY.md, supersedes the old "11
separate Workers/Durable Objects" framing):** a single engine running in
one Cloudflare Worker (`agent-runner.js`) role-plays all 11 personalities
defined in `config/agents-config.json`. Nothing from the original
per-agent design (configs, year-tracker, side-plot/promotion tracks,
admin dashboard) is deleted — it stays in place as the data layer/spec
the single engine reads from and acts on.

**Sequencing (current priority order, per STRATEGY.md):**
1. UI polish — `index.html` excellent, fast, mobile-ready, bilingual,
   working Claude AI Search end-to-end.
2. Consolidate the agent runtime into the single engine (re-architecture
   of existing code, not new feature work).
3. Test the single engine against the live app.
4. Run the full simulation.

**Do not (for now):** stand up additional per-agent Workers/Durable
Objects; build a separate behavioral engine for agents 5–11 (they use the
same single-engine, config-driven approach as agents 1–4).

## 5. Cases

- 20 cases/agent/day. ~10 active agents → roughly 180–200 cases/day
  total (Architect, agent 10, never touches an AI model for routine
  cases).
- Generated by `crm-engine.js` / `case-generator.js` — Netvill-flavored
  originally, now scoped to general IT + cybersecurity to match the
  app's expanded remit.
- Each agent only escalates to Claude when case difficulty + personality
  state trigger it — not every case touches the model.

## 7. The 11 agents

### Workers (dedicated state machines, standard clearance)

1. **The Perfectionist** — QA lead persona. Uses Claude for complex
   cases/syntax checks, deep-dives articles, curates bookmarks, extends
   sessions 50% when satisfied. Never angry — lectures rather than rages
   when the model is wrong. Mood: Happy → uses Claude more · Irritated →
   critical/corrective.
2. **The Productive** — efficiency-focused. Works overtime 30% of days,
   earns a bonus day at >30% over weekly target. Irritation stacks —
   three unresolved frustrations trigger an incident report and exit.
3. **The Standard Agent** — the control group. Claude usage scales
   0–100% directly with satisfaction. Always issues a balanced status
   report after a session.
4. **The Trainee** — documentation-dependent, panics under pressure
   (40% calls QA, 30% Perfectionist, 30% another agent for help — the
   teammate joins the session and writes a guide if none exists). Happy
   → 40% productivity boost for 5 sessions.

### Admins (SUDO / specialist clearance, generic `agent-stub.js` driver)

5. **The IT Chief (SUDO)** — senior troubleshooter, hard client
   escalations, network optimization. Mishandled clients trigger private
   coaching; too many unhappy clients trigger a team huddle. Happy →
   300% productivity boost.
6. **The QA (SUDO)** — weekly audits alongside the Team Lead, sampling
   random cases from each agent's prior week, rating model vs. agent.
   Inherits Perfectionist traits (deep-dives, bookmarks, lectures).
7. **The Team Lead (SUDO)** — agent development/productivity coach. Can
   place agents 1–4 on a 1-month PIP. Sits in on QA audits. Generates
   high morale for the first 2 days of each week.
8. **The Lead QA (SUDO)** — chief quality officer, broadest audit scope
   (agents, workflows, technologies, the model itself). Mood moves
   slowly. Amplifies personality-trigger frequency for everyone present
   when he speaks in a meeting.
9. **The Designer (Specialist)** — focused on the GitHub repo and app
   UI. 2 reports/week. Starts at 0% fondness, becomes **Inspired** at
   51% (artistic capabilities +300%). 4 major quarterly UI updates.

### Management (ROOT clearance)

10. **The Architect (ROOT, no AI model)** — knows every corner of the
    product. Never calls an AI for routine cases — cases are filed as a
    single GitHub Issue (`claude-action`, `architect-task`) for
    human/Claude Code review. Releases numbered update packages (v1.0,
    v1.1...). Combines Perfectionist + Productive + IT Chief traits.
    Professional rivalry with Lead QA.
11. **The CEO (ROOT)** — leads all meetings, votes count double. Creates
    a zone of influence: everyone in joint sessions gets +20%
    morale/productivity. 20% of her cases are unique high-value clients
    whose outcomes shape her model opinion.

Shared machinery: each agent has a `mood`/`irritation`/`isPanic` state
machine (`agents/agent-base.js`), durable per-agent overrides in its
`AgentStateDO`, and a `clearance` tier (`standard` < `specialist` <
`sudo` < `root`) that routes `fileSuggestion()` calls.

## 8. Permissions

- **SUDO** — minor/medium changes independently; consults the admin team
  for bigger ones.
- **ROOT (Architect)** — medium changes independently; needs CEO
  approval for large/radical changes; final say on fundamental changes.
- Rivalries/side-plots are enabled but constrained to work-related
  content intended to inspire improvement, not conflict.

## 9. Reports

Daily → Weekly (Friday, PDF + Excel) → Monthly → Quarterly. Semi-yearly/
yearly cadence has been dropped (doesn't fit the one-day-at-a-time
execution model — revisit only if the simulation runs long enough for it
to matter again). Yearly-deliverable framing in README.md (12-month
Excel, 10-page executive PDF, etc.) is legacy and superseded by this.

## 11. How to deploy the Worker

```bash
npx wrangler login   # or set CLOUDFLARE_API_TOKEN
npx wrangler secret put GEMINI_API_KEY
npx wrangler secret put GROQ_API_KEY
npx wrangler secret put ADMIN_TOKEN
npx wrangler secret put GITHUB_TOKEN       # optional
npx wrangler secret put GOOGLE_AI_API_KEY  # optional
npx wrangler deploy
```

Verify via `/api/agents/status`. Full walkthrough in `DEPLOY.md`.

## 12. How to trigger a simulation day manually

```bash
curl -X POST https://data-center-agents.avivnofar.workers.dev/api/agents/trigger \
  -H "X-Admin-Token: <ADMIN_TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{"type":"day"}'
```

Other trigger types: `meeting` (`{"meetingType": "..."}`), `inspection`
(`{"active": true|false}`), `week_reset`. See `agent-runner.js`'s
`/api/agents/trigger` handler for the full switch.
